🔧 04 — Setup: Auth & Prerequisites

⚠️ Critical: Traffic API endpoints require push access to the repo. Read-only tokens will get a 403 Forbidden. This means you need either:
  • A GitHub PAT (classic) with repo scope
  • OR: Be logged into gh CLI with an account that has push access

Option A: GitHub CLI (Recommended)

If you're logged into the gh CLI with push access to the repo:

# Check current auth
gh auth status

# If you need to switch accounts:
gh auth switch --user denise-msft

# Set the token for scripts:
$env:GITHUB_TOKEN = (gh auth token)

Option B: Personal Access Token (PAT)

1

Create a Classic PAT

Go to github.com/settings/tokens → Generate new token (classic)

2

Select the repo scope

This grants push-level access needed for traffic endpoints. The repo scope includes repo:status, repo_deployment, public_repo, and repo:invite.

3

Set the environment variable

# PowerShell (session)
$env:GITHUB_TOKEN = "ghp_your_token_here"

# Or add to your profile for persistence:
# Add to $PROFILE: $env:GITHUB_TOKEN = "ghp_..."
🔑 PAT vs OAuth: gh auth token returns an OAuth token that may lack repo scope. If traffic endpoints return 403, create a classic PAT with explicit repo scope instead.

⚡ 09 — Known Gotchas & Troubleshooting

🚫 403 Forbidden on Traffic Endpoints
Cause: Your token lacks push access to the repo.
Fix: Create a classic PAT with repo scope, OR ensure your gh CLI account has push access.
⏰ 14-Day Rolling Window
Traffic data (views, clones, paths, referrers) only covers the last 14 days. If you don't collect it, it's gone forever.
Fix: Run the Fabric notebook weekly to persist historical data.
🔄 gh auth token May Lack Scope
The OAuth token from gh auth token may not include repo scope for Microsoft org repos.
Fix: Create a classic PAT at github.com/settings/tokens with explicit repo scope.
📊 Static HTML Dashboard
The dashboard.html has data baked in at generation time. To refresh it, re-run the PowerShell script that generates it.
🔀 GitHub Account Switching
The default active GH account may be demora_microsoft (Microsoft org). For repos under denise-msft, run gh auth switch --user denise-msft first.
🏢 Kusto VPN Requirement
All fdislands* Kusto clusters require Microsoft corpnet. If queries fail, check your VPN connection.
🔐 Code Apps CSP Blocks Direct Telemetry
The Managed Host iframe enforces connect-src 'none' — you cannot send telemetry directly from a Code App to 1DS/Aria. Must route through Dataverse Custom APIs instead.

🎯 10a — Download Tracking: The Critical Gap

⚠️ The #1 thing to understand: GitHub tracks three download methods completely separately, and one of them isn't tracked at all. If you don't route downloads correctly, you'll lose visibility.

What GitHub Actually Tracks (and Doesn't)

Download MethodUser ActionTracked?APIRetention
git clone User runs git clone in terminal ✅ Yes Traffic API /traffic/clones Rolling 14 days
Release Asset Download User clicks a file attached to a GitHub Release ✅ Yes Releases API /releases → download_count Cumulative (forever) ✅
"Download ZIP" button User clicks green Code button → Download ZIP ❌ NOT tracked None N/A
Raw file link User clicks a .zip file inside the repo tree ❌ NOT tracked None N/A
raw.githubusercontent.com Direct link to blob content ❌ NOT tracked None N/A
🚨 This means: The .zip files currently sitting inside each template folder (e.g., Plan My Day/PlanMyDay_v1.0.0.0.zip) are NOT trackable as downloads. When someone clicks them in the GitHub UI or follows a raw link, that download is invisible to us. We can only see them in popular/paths page views, not actual downloads.

The Fix: Use GitHub Releases

GitHub Releases are the only GitHub-native mechanism that gives per-file download counts. The strategy:

1

Create a tagged Release per template

Each template gets its own release tag (e.g., plan-my-day-v1.0.0) with the .zip uploaded as a release asset.

# Create releases via gh CLI:
gh release create plan-my-day-v1.0.0 \
  "Plan My Day/PlanMyDay_v1.0.0.0.zip" \
  --repo microsoft/m365-agent-templates \
  --title "Plan My Day v1.0.0" \
  --notes "Declarative Agent for daily briefing from calendar, email, Teams, and files."

gh release create my-company-policy-v1.0.0 \
  "My Company Policy/MyCompanyPolicy_1_0_0_0.zip" \
  --repo microsoft/m365-agent-templates \
  --title "My Company Policy v1.0.0" \
  --notes "Declarative Agent for policy Q&A grounded in SharePoint documents."

gh release create executive-briefing-v1.0.0 \
  "Executive Briefing/ExecutiveBriefingAgent_v1.0.0.0.zip" \
  --repo microsoft/m365-agent-templates \
  --title "Executive Briefing v1.0.0" \
  --notes "Declarative Agent for meeting prep and executive briefings."

gh release create request-tracker-v1.0.0 \
  "Request Tracker/RequestTrackerAgent_1_0_0_0.zip" \
  --repo microsoft/m365-agent-templates \
  --title "Request Tracker v1.0.0" \
  --notes "Custom Agent (Copilot Studio) for internal request management."

gh release create know-my-customer-v1.0.0 \
  "Know My Customer/KnowMyCustomer_1_0_0_0.zip" \
  --repo microsoft/m365-agent-templates \
  --title "Know My Customer v1.0.0" \
  --notes "Custom Agent (Copilot Studio) for 360° customer profiles."
2

Point ALL download links to the Release asset

Every download path — aka.ms links, SharePoint links, internal docs, field decks — must route to the same Release asset URL:

# Release asset URL pattern:
https://github.com/microsoft/m365-agent-templates/releases/download/{tag}/{filename}

# Examples:
https://github.com/microsoft/m365-agent-templates/releases/download/plan-my-day-v1.0.0/PlanMyDay_v1.0.0.0.zip
https://github.com/microsoft/m365-agent-templates/releases/download/request-tracker-v1.0.0/RequestTrackerAgent_1_0_0_0.zip

Then create aka.ms short links that redirect to these:

aka.ms/cskagentlib/dailyplanner    → releases/download/plan-my-day-v1.0.0/PlanMyDay_v1.0.0.0.zip
aka.ms/cskagentlib/companypolicy   → releases/download/my-company-policy-v1.0.0/MyCompanyPolicy_1_0_0_0.zip
aka.ms/cskagentlib/execbriefing    → releases/download/executive-briefing-v1.0.0/ExecutiveBriefingAgent_v1.0.0.0.zip
aka.ms/cskagentlib/requesttracker  → releases/download/request-tracker-v1.0.0/RequestTrackerAgent_1_0_0_0.zip
aka.ms/cskagentlib/knowmycustomer  → releases/download/know-my-customer-v1.0.0/KnowMyCustomer_1_0_0_0.zip
3

Version updates preserve history

When you release v1.1.0, create a new release tag (e.g., plan-my-day-v1.1.0). The old release keeps its download count. Update the aka.ms link to point to the new release. This gives you:

  • Per-version download counts — see which version was most popular
  • Total all-time downloads — sum across all releases
  • Preserved history — old counts never disappear

The Routing Rule

🔗 One Download Link to Rule Them All

Every place that links to a template download — README, SharePoint, internal docs, field decks, Teams messages, email signatures — MUST point to the same GitHub Release asset URL (or an aka.ms redirect to it). If someone copies the .zip to SharePoint and shares that link instead, those downloads are invisible.

Source✅ Do This❌ Not This
README.md Link to Release asset or aka.ms redirect Link to blob in repo tree
SharePoint page Link to aka.ms → Release asset Upload .zip to SPO document library
Field deck / email Link to aka.ms → Release asset Attach .zip directly
Agent Library app "Download" button → Release asset URL Serve from Dataverse/blob

What We CAN Track Separately

SignalAPIWhat It Tells YouRetention
Clones (git clone) /traffic/clones Developers pulling the full repo to customize/contribute 14 days (rolling)
Release downloads /releases → per-asset download_count Users downloading a specific template package to deploy Forever (cumulative) ✅
Page views /traffic/views People visiting the repo (awareness/discovery) 14 days (rolling)
Popular paths /traffic/popular/paths Which template folders are getting the most attention 14 days (rolling)
Referrers /traffic/popular/referrers Where traffic is coming from (aka.ms, Teams, Google, etc.) 14 days (rolling)
💡 The "Download ZIP" button (green Code button → Download ZIP) downloads the entire repo as a zip. This is NOT counted anywhere — not in clones, not in releases, not in traffic. There is no GitHub API for this. If users are told to "download the whole repo," those downloads are invisible. Always point them to specific Release assets instead.

Querying Release Download Counts

# PowerShell — get per-template download counts
$headers = @{ Authorization = "Bearer $env:GITHUB_TOKEN"; Accept = "application/vnd.github+json" }
$releases = Invoke-RestMethod "https://api.github.com/repos/microsoft/m365-agent-templates/releases" -Headers $headers

foreach ($r in $releases) {
    $total = ($r.assets | Measure-Object download_count -Sum).Sum
    Write-Host "$($r.tag_name): $total downloads"
    foreach ($a in $r.assets) {
        Write-Host "  $($a.name): $($a.download_count)"
    }
}

Global Cluster Union Pattern (Required for Complete Data)

Always union all 11 regional clusters — querying a single cluster only returns data for that region:

let _OrgSolutionsInstalled = OrgSolutionsInstalled
  | union cluster('fdislandswebeuams.westeurope.kusto.windows.net').database('CRMAnalytics').OrgSolutionsInstalled
  | union cluster('fdislandswebeudb3.northeurope.kusto.windows.net').database('CRMAnalytics').OrgSolutionsInstalled
  | union cluster('fdislandswebausyd.australiaeast.kusto.windows.net').database('CRMAnalytics').OrgSolutionsInstalled
  | union cluster('fdislandswebjptyo.japaneast.kusto.windows.net').database('CRMAnalytics').OrgSolutionsInstalled
  | union cluster('fdislandswebcayto.canadacentral.kusto.windows.net').database('CRMAnalytics').OrgSolutionsInstalled
  | union cluster('fdislandswebfrfrc.francecentral.kusto.windows.net').database('CRMAnalytics').OrgSolutionsInstalled
  | union cluster('fdislandswebdegec.germanywestcentral.kusto.windows.net').database('CRMAnalytics').OrgSolutionsInstalled
  | union cluster('fdislandswebchzrh.switzerlandnorth.kusto.windows.net').database('CRMAnalytics').OrgSolutionsInstalled
  | union cluster('fdislandswebnoeno.norwayeast.kusto.windows.net').database('CRMAnalytics').OrgSolutionsInstalled
  | union cluster('fdislandswebsecse.swedencentral.kusto.windows.net').database('CRMAnalytics').OrgSolutionsInstalled;
⚠️ KQL Syntax Gotcha: You CANNOT use let db = cluster(...).database(...); db.Table — that syntax is invalid. Always use the full inline cluster('...').database('...').Table.

Agent Template Install Count

// Count installs per agent template across all regions
_OrgSolutionsInstalled
| where SolutionUniqueName startswith "AgentLib_"
| where PublisherUniqueName == "PowerCat"
| summarize Environments = dcount(OrganizationId) by SolutionUniqueName, SolutionVersion
| order by Environments desc

Agent Template Usage (Durable via Bot Schema)

ChatbotDetails
| where BotSchemaName in (
    "cat_dailyplanner", "cat_customerresearcher", "cat_executiveinsights",
    "cat_companypolicyqa", "cat_requesttracker"
)
| summarize Environments = dcount(OrganizationId), Bots = count() by BotSchemaName
| order by Environments desc

Tenant Name Resolution

// Join org → tenant → company name
_OrgSolutionsInstalled
| where SolutionUniqueName startswith "AgentLib_"
| join kind=inner OrganizationDetails on $left.OrganizationId == $right.Id
| join kind=leftouter (
    cluster('powerautomatefunbipme.eastus2.kusto.windows.net')
      .database('SelfServiceData').dimTenant
    | where IsTestTenant == 0
    | project TenantId, TenantName
) on TenantId
| project SolutionUniqueName, TenantName, FriendlyName, DomainName, Geo, CountryCode
